## 2. 현재 사이트 상태 (검증된 사실)

저장소: https://github.com/jongwony/jongwony — Vue 3 + Bootstrap 5 단일 페이지 앱, GitHub Pages 배포.

**구조상 알아야 할 점**

- 사이트 전체 화면이 `src/components/HelloWorld.vue` 한 컴포넌트에 들어 있다(240줄). 섹션 7개(헤더·기술 태그·언론 스크랩·발표·사이드 프로젝트·GitHub·연락처)가 이 한 파일에서 렌더링된다.
- 콘텐츠 데이터는 빌드에 포함되지 않고 **런타임에 GitHub raw URL에서 fetch**된다(`src/components/HelloWorld.vue` 145–170행 부근). 그래서 JSON만 고쳐 main에 push하면 사이트에 바로 반영되지만, 초기 HTML에는 콘텐츠가 0바이트라 **검색엔진·링크 미리보기 봇에게는 사이트 내용이 보이지 않는다.**
- fetch 실패 시 에러 처리가 없어 해당 섹션이 조용히 비어 버린다.
- 서체는 이미 Pretendard Variable을 쓴다(`src/App.vue` 20행에 폰트 스택, `public/index.html` 9행에 CDN 로드).
- 기존 액센트 색 `#2178FC`가 `src/components/HelloWorld.vue` 215행에 있다.

**콘텐츠 재고** (2026-07-19 기준, 위 검증 명령으로 확인 가능)

| 데이터 파일 | 건수 | 내용 | 배치 설계에 걸리는 점 |
|---|---|---|---|
| `src/data/speaker.json` | 21 | 발표·강연 이력, 2017-10-21 ~ 2026-01-24 | 21건 중 링크가 있는 것이 17건, 그중 **유튜브 영상 링크는 5건**. 배열은 이미 최신순(최근 것이 위). |
| `src/data/scrap.json` | 9 | 언론 노출·대외활동 (2018–2024) | 뉴스 이미지가 외부 사이트 직접 링크라 링크 부패 위험 |
| `src/data/bio.json` | 5 | 사이드 프로젝트 | GitHub 저장소명과 매칭될 때만 카드가 뜨는 구조라, 이름이 안 맞는 항목은 카드가 깨져 있을 가능성이 높다 |
| `src/data/tags.json` | 1 | 기술 스택 배지 | 사실상 비어 있음 |

각 파일의 필드 구조는 저장소 루트 `CLAUDE.md`의 "speaker.json Schema" / "bio.json Schema" 절에 적혀 있다.

## 3. 이미 확정된 것 — 다시 열지 말 것

전문은 Linear의 **디자인 결정 · 토큰 v1** 문서에 있다(위 허용 소스). 배치 설계에 직접 걸리는 것만 여기 옮긴다.

**확정된 사이트 형태**

첫 화면은 **어두운 배경의 영상 히어로** — 소유자의 실제 유튜브 발표 영상에서 딴 클립이 무음으로 반복 재생되고, 그 위에 한 줄 헤드라인이 얹힌다. 스크롤을 내리면 **밝고 정적인 본문**으로 전환되며, 본문은 활동을 이야기로 풀어내는 **연대기**다. 참고한 형태는 harrisonwheeler.com/about (포지셔닝 헤드라인 → 가치 몇 가지 → 연도가 앵커인 "나의 이야기" 연대기)이고, 그 페이지가 사진으로 각 챕터를 앵커하는 자리를 이 사이트에서는 영상 클립이 대신한다.

**확정 좌표 (요약)**

1. 히어로 매체 = 실제 유튜브 클립 무음 자동재생 루프
2. 모션 무게중심 = 히어로에만. 본문은 정적 기조
3. 서체 = Pretendard 계승
4. 액센트 = 기존 파랑 `#2178FC` 계열 계승
5. 챕터 미디어 = **혼합** — 클립이 있으면 썸네일, 사진이 있으면 사진, 없으면 텍스트만. **모든 챕터가 미디어를 갖지는 않는다**
6. 진행 순서 = 디자인 토큰 → 컴포넌트 → 페이지 (토큰은 이미 확정됨)
7. 시각 반복 = 브라우저 스크린샷을 보고 고치는 루프를 표준 작업 방식으로
8. 본문 스크롤 모션 = 챕터가 화면에 들어올 때 한 번 부드럽게 나타나는 정도까지만. 스크롤에 연동해 계속 움직이는 연출(패럴랙스·고정)은 배제
9. 라이브러리 = 웹 표준(CSS, IntersectionObserver) 우선. 표준으로 안 되는 연출만 경량 라이브러리를 개별 채택. 무거운 3D·차트 라이브러리는 명확한 용처가 생길 때만
10. **연대기 정렬 = 내림차순.** 최신이 맨 위, 아래로 갈수록 과거

디자인 토큰 v1(색·타이포·간격·모션 값 전량)은 Linear 결정 문서 §4에 CSS 커스텀 프로퍼티 형태로 그대로 실려 있다. 접근성 관련 판단 하나가 이미 내려져 있다: `#2178FC`는 흰 배경 대비 4.07:1이라 본문 크기 텍스트 기준(4.5:1)에 못 미치므로, 본문 링크에는 더 진한 `#0F5FE3`(5.6:1)을 쓰고 원래 파랑은 버튼·아이콘·큰 글자에만 쓴다.

**전제로 깔린 것**: 사용자가 모션 최소화를 설정했거나 모바일인 경우 히어로는 정지 화면으로 대체된다. 자동재생은 무음이어야 한다.

## 4. 다음 작업의 출발점과 정할 것들

**출발점 (소유자 발언)**: "연대기 챕터를 내림차순으로 위에서 아래로 보여주는 것만 있어도 시작하기에 좋겠다." 즉 완결된 정보 구조를 한 번에 설계할 필요는 없고, **연대기 하나가 최소 출발 형태**다.

정해야 할 것:

1. **섹션 구성** — 현재 사이트의 7개 섹션(헤더·기술 태그·언론·발표·사이드 프로젝트·GitHub·연락처) 중 무엇이 살아남고, 무엇이 연대기로 흡수되고, 무엇이 사라지는가. 발표 21건·언론 9건·프로젝트 5건을 하나의 연대기로 합칠지, 종류별로 나눌지.
2. **연대기 항목의 단위와 밀도** — 21건 전부가 한 챕터씩 차지하는지, 묶이는지. 연도 헤더를 두는지. (참고: 유튜브 클립이 붙을 수 있는 항목은 5건뿐이므로 대부분의 항목은 텍스트만이다 — 좌표 5의 "혼합"이 실제로 어떤 비율이 되는지 여기서 드러난다.)
3. **외부 글 수집 경로** — LinkedIn·Threads·GitHub 릴리스 노트에 있는 글을 사이트로 가져오는 방법. 수동 복사인지, 피드 기반인지, 링크만 모으는지. 이건 **콘텐츠 운영 방식 결정**이라 배치 설계와 맞물린다.
4. **초기 HTML에 콘텐츠가 없는 문제를 배치가 어떻게 감당할지** — 현재 구조에서는 크롤러에 아무것도 안 보인다(§2). 해결 자체는 기술 스택 결정(§5 미결)에 딸리지만, 배치 설계가 그 결정에 제약을 건다.

## 5. 미결 사항 (이 작업에서 결론 낼 필요는 없지만 알고 있어야 함)

- **기술 스택** — 렌더링 방식(정적 생성/서버 렌더링 여부)과 프레임워크 유지/전환이 정해지지 않았다. 현재는 Vue 3. React 전용 컴포넌트 라이브러리를 쓰고 싶다는 얘기가 나왔는데, 그건 프레임워크 전환이 전제라 이 결정에 종속된다.
- **영상 히어로의 실행 조건** — 실제 유튜브 임베드의 자동재생 정책, 초기 로딩 비용, 모바일 처리를 아직 실측하지 않았다. 구현 착수 시점에 확인할 항목.
- **사이드 프로젝트 카드 깨짐** — `src/data/bio.json`의 항목 중 GitHub 저장소명과 안 맞는 것들의 카드가 깨져 있을 가능성이 높다(미확인). 배치에서 이 섹션을 유지한다면 확인이 필요하다.

## 6. 작업 규칙 (이 저장소에서 지켜야 할 것)

- **에이전트가 수행하는 파일 편집은 격리된 worktree의 feature 브랜치에서 하고 PR로 올린다.** main에 직접 커밋하지 않는다. 병합은 사람이 웹에서 직접 한다.
- **Linear에는 결정만 기록한다.** 방향·좌표·확정 값 같은 결정은 프로젝트 문서나 코멘트로 남기되, 작업 단위(이슈 트리·의존 관계)나 세션 간 인계는 Linear에 넣지 않는다. 상태·진행률은 손으로 쓰지 않는다.
- Linear 프로젝트: [jongwony.com 전면 리뉴얼](https://linear.app/rootproto/project/jongwonycom-전면-리뉴얼-0998766eac43) (rootproto 워크스페이스, RootProto 팀).
- **이 작업의 결과를 기록할 곳**: 위 Linear 프로젝트의 "디자인 결정 · 토큰 v1" 문서. 배치 결정이 정해지면 그 문서에 절을 추가하거나 새 결정 문서를 만든다.
